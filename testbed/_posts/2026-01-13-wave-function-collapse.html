What is Wave Function Collapse?

Wave function collapse is a quantum mechanics concept where a system in a superposition of states instantly reduces to a single, definite state upon measurement or observation. In computer science, the Wave Function Collapse (WFC) algorithm is a technique for generating procedural, constraint-based content, such as tilemaps or textures, similar to solving a 3D Sudoku puzzle.

Each cell in a grid is initially treated as a superposition of all possible states. Rather than sampling all cells simultaneously, WFC proceeds by selecting a single cell, collapsing its state space to one concrete value, and then propagating the resulting constraints to neighboring cells.

Let's first look at the 3x3 block in the right-center of the leftmost image. Each cell in this area has two possible candidate numbers, and the red cells all share the same candidates ($4, 6$). Because the red cells must be filled with either $4$ or $6$, the state of the blue cell is forced to collapse to $2$ otherwise, a contradiction would occur. After the blue cell collapses to $2$, the cell directly above it, which has $2$ and $7$ as candidates, now collapses to $7$.


Tileset Preparation

To apply WFC to real-world design, the abstract states must be turned into concrete, connectable blocks. The algorithm then operates on a finite set of such modules. Each module is defined by how it can attach to its neighbors, so that collapse and propagation are purely rule-based. A useful reference is modular systems, where a small set of physical pieces combine in many ways under clear connection rules. LYCS's Modular CATable is one example. A limited set of table segments with defined interfaces can be rearranged into different table layouts. That idea translates directly into a 3D tileset for WFC.

Inspired by this kind of modular design, I defined the 3D tileset for WFC. Each tile is a block with six faces ($\pm x$, $\pm y$, $\pm z$), and each face has an interface label. Two neighboring tiles are compatible only if their touching faces have the same compatible labels (6-way compatibility). The algorithm uses these rules, not geometry, to propagate constraints.

The left panel on the figure below shows the 3D tileset, and the right panel shows the compatibility visualization. The right panel explains that the six dots per cube represent one label per face, color-coded, and that allowed adjacencies come from compatible labels on touching faces. For example, the tile $6$ and $7$ are compatible for the $+z$ and $-z$ faces each other. Because the $+z$ face of $6$ is OPEN and the $-z$ face of $7$ is OPEN, they are compatible.

In code, the tileset is stored as a Python dictionary INTERFACE: each key is a tile ID from $0$ to $19$, and each value is a mapping from the six face directions ($\pm x$, $\pm y$, $\pm z$) to one of five labels—FREE, OPEN, CLOSED, SUPPORT, and EMPTY. Tile $0$ is the empty cell. Tiles $1$–$19$ are the solid building blocks. Compatibility between neighboring tiles is determined by these labels: for any pair of touching faces, the labels must match, except that FREE acts as a wildcard and is compatible with any label on the adjacent face.
